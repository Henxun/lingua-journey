## Context

Lingua Journey 目前已实现基础的认证系统，包括 Google/GitHub OAuth 登录和邮箱验证码登录/注册。然而，缺少两个重要的用户功能：

1. **账户设置**：用户无法查看和管理已关联的登录方式（OAuth 账户绑定/解绑），也无法修改个人偏好设置
2. **忘记密码**：当用户忘记密码时，无法通过邮箱验证重置密码

这两个功能对于提升用户体验和账户安全性至关重要。

**现有系统组件：**
- 后端：`authService.ts`、`emailService.ts`、`oauthService.ts`
- 前端：`AuthContext.tsx`、登录/注册页面
- 数据库：User 实体已包含 `oauth_profiles` JSON 字段
- 邮件系统：已配置 Nodemailer，支持验证码邮件

## Goals / Non-Goals

**Goals:**
- 提供账户设置页面，让用户管理已关联的 OAuth 账户
- 实现安全的忘记密码流程，通过邮箱验证码重置密码
- 确保账户安全：解绑 OAuth 需要验证身份，至少保留一种登录方式
- 提供一致的 UI/UX 体验，与现有认证页面风格统一

**Non-Goals:**
- 不实现双因素认证（2FA）
- 不实现社交分享功能
- 不支持手机号登录/验证
- 不实现企业级 SSO（Single Sign-On）

## Decisions

### Decision 1: OAuth 账户关联流程

**选择方案：** 用户通过 OAuth 授权码关联账户，需要输入现有邮箱账户的密码验证身份

**理由：**
- 安全性高：只有知道邮箱密码的用户才能关联新 OAuth
- 用户体验好：直接跳转授权，无需额外验证步骤
- 避免冲突：同一邮箱不能被多个 OAuth 账户占用

**备选方案：**
- 邮箱验证码验证：用户需要能接收邮件，但可能无法登录
- OAuth 直接合并：如果 OAuth 邮箱与现有邮箱相同，直接关联（风险：可能误关联他人账户）

### Decision 2: OAuth 账户解绑限制

**选择方案：** 用户解绑 OAuth 账户时，如果只剩一种登录方式，阻止解绑操作

**理由：**
- 账户安全：防止用户将自己锁在账户外
- 兜底机制：确保至少有一种登录方式可用

**备选方案：**
- 无限制解绑：用户可能将自己锁在账户外（风险太高）
- 强制绑定新方式后才能解绑：流程复杂，用户体验差

### Decision 3: 忘记密码流程

**选择方案：** 使用现有的邮箱验证码系统，通过 `VerificationCodeType.RESET_PASSWORD` 类型区分

**理由：**
- 复用现有基础设施：不需要新建验证码实体
- 流程一致：与注册/登录验证码流程统一
- 安全性：15分钟有效期，限制重试次数

**备选方案：**
- 邮箱链接验证：用户点击链接重置密码（更安全但实现复杂）
- 人工客服验证：安全性高但成本高，不适合自动化系统

### Decision 4: 前端路由设计

**选择方案：** 
- `/profile/settings` - 账户设置页面
- `/forgot-password` - 忘记密码页面
- 在登录页面添加"忘记密码"链接

**理由：**
- 符合 Next.js 文件路由规范
- URL 语义清晰，用户易理解
- 与现有 `/login`、`/register` 页面风格一致

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| OAuth 关联时邮箱已被占用 | 用户无法关联账户 | 提示用户邮箱已被占用，建议登录后关联 |
| 解绑后只剩密码登录 | 如果密码也忘记，将无法登录 | 在解绑页面添加"忘记密码"引导 |
| 邮件发送失败或延迟 | 用户无法完成验证 | 提供控制台验证码（开发模式）+ 明确的错误提示 |
| 暴力破解验证码 | 安全性风险 | 验证码 6 位数字，限制请求频率（60秒/次），15分钟过期 |
| CSRF 攻击 | 账户设置被恶意修改 | 使用 JWT token 验证身份，所有请求需要认证 |

## Migration Plan

**Phase 1: 后端 API 开发（预计工作量：小）**
1. 扩展 `authController.ts`：新增账户设置和密码重置端点
2. 扩展 `authRoutes.ts`：注册新路由
3. 扩展 `authService.ts`：新增业务逻辑方法
4. 测试所有 API 端点

**Phase 2: 前端页面开发（预计工作量：中）**
1. 创建 `/profile/settings` 页面
2. 创建 `/forgot-password` 页面
3. 更新 `AuthContext.tsx`：新增相关方法
4. 更新 `api.ts`：新增 API 调用
5. 集成到现有导航和页面流程

**Phase 3: 测试和部署（预计工作量：小）**
1. 功能测试：账户设置、密码重置全流程测试
2. 安全测试：CSRF、暴力破解、权限验证
3. 部署：合并到主分支，生产环境测试
4. 监控：添加日志和错误追踪

**回滚策略：**
- 如果 Phase 3 发现严重问题，可通过 git revert 回滚
- 新功能独立，不影响现有登录/注册功能
